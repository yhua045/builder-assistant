## 1. User Story

> As an owner-builder starting a new project,  
> after I enter the project details and the project is saved,  
> I want to be shown a list of suggested critical-path stages (all ticked by default)  
> so that I can deselect any I don't need and click "Add N Tasks to Plan"  
> to immediately scaffold a realistic construction sequence.

### 3c. Application — Idempotent `CreateTaskUseCase` / hook strategy

**Design decision:** Do NOT change `CreateTaskUseCase` itself. Instead, `useCriticalPath.confirmSelected()` will use a **stable pre-allocated UUID** per suggestion:

- When `suggest()` resolves, each `CriticalPathSuggestion.id` (e.g. `"cp-NSW-01"`) is used as a **seed** to generate a deterministic task UUID via a `stableId(projectId, suggestionId)` helper (SHA-1-style prefix, no external dep — use template string hash).
- `confirmSelected` stores a `Map<suggestionId, taskId>` in hook state. On the first call, IDs are generated and stored. On **retry**, the same IDs are reused.
- On retry, `DrizzleTaskRepository.save()` already checks `findById` and calls `update()` if the row exists — so re-submitting the same ID is a no-op for already-created tasks, and the failed tasks are retried with their original IDs.

**New utility:** `src/utils/stableId.ts`  
```typescript
export function stableId(projectId: string, suggestionId: string): string {
  // Non-cryptographic stable hash for deterministic task IDs.
  // Produces a UUID-shaped string that is stable across retries.
  const raw = `cp:${projectId}:${suggestionId}`;
  let h = 0x811c9dc5;
  for (let i = 0; i < raw.length; i++) {
    h ^= raw.charCodeAt(i);
    h = (h * 0x01000193) >>> 0;
  }
  const hex = h.toString(16).padStart(8, '0');
  return `cp-${hex}-${projectId.slice(0, 8)}-${suggestionId.slice(0, 8)}`;
}
```

**`useCriticalPath` hook update:** Add a `taskIdMap: Map<string, string>` ref (not state — no re-render). Populate it in `suggest()` once suggestions arrive. `confirmSelected` reads from the map to build the task `id` before calling `CreateTaskUseCase.execute(...)`.

> **Constraint:** `CreateTaskUseCase.execute` currently takes `Omit<Task, 'id' | 'createdAt' | 'updatedAt' | 'localId'>`. To pass a pre-allocated `id`, either:
> - (Option A) Add an optional `id?: string` parameter to the use case, OR
> - (Option B) Pass the task directly to `taskRepository.save()` from the hook, bypassing the use case.
>
> **Decision: Option A** — add `id?: string` to `CreateTaskUseCase` params. If provided, use it; otherwise generate a UUID. This keeps the architecture clean.

### 3d. UI — ManualProjectEntry (multi-step flow)

**File:** `src/components/ManualProjectEntry.tsx`

Replace the single-step `ManualProjectEntryForm` + Alert pattern with a two-step flow:

```typescript
type Step = 'form' | 'critical-path';

const ManualProjectEntry: React.FC<Props> = ({ initialVisible = false }) => {
  const [visible, setVisible] = React.useState(initialVisible);
  const [step, setStep] = React.useState<Step>('form');
  const [newProjectId, setNewProjectId] = React.useState<string | null>(null);
  const [criticalPathRequest, setCriticalPathRequest] =
    React.useState<SuggestCriticalPathRequest | null>(null);

  const { createProject } = useProjects();
  // DI: useCriticalPath instantiated here with use cases from DI container

  const handleSave = async (dto: CreateProjectRequest & { projectType: ProjectType; state?: AustralianState }) => {
    const result = await createProject(dto);
    if (result.success && result.projectId) {
      setNewProjectId(result.projectId);
      setCriticalPathRequest({ project_type: dto.projectType, state: dto.state });
      setStep('critical-path');
    } else {
      Alert.alert('Error', result.errors?.join('\n') ?? 'Failed to create project');
    }
  };

  const handleTasksConfirmed = () => {
    setVisible(false);
    setStep('form');
    setNewProjectId(null);
  };

  return (
    <View>
      <ManualProjectEntryButton onPress={() => setVisible(true)} />
      <Modal visible={visible} animationType="slide" presentationStyle="fullScreen">
        {step === 'form' ? (
          <ManualProjectEntryForm
            onSave={handleSave}
            onCancel={() => { setVisible(false); setStep('form'); }}
          />
        ) : (
          <CriticalPathStep
            projectId={newProjectId!}
            request={criticalPathRequest!}
            onDone={handleTasksConfirmed}
            onSkip={handleTasksConfirmed}
          />
        )}
      </Modal>
    </View>
  );
};
```

**New sub-component:** `CriticalPathStep` (thin wrapper, lives in `ManualProjectEntry.tsx` or extracted to `src/components/CriticalPathPreview/CriticalPathStep.tsx`)  
- Instantiates `useCriticalPath` with DI-resolved use cases
- Calls `suggest(request)` on mount via `useEffect`
- Passes `hookResult` to existing `CriticalPathPreview`
- Provides "Skip" link (secondary link, non-prominent) so users can bypass task creation

### 3e. UI — ManualProjectEntryForm (project type + state fields)

**File:** `src/components/ManualProjectEntryForm.tsx`

Add two new fields to the form (and to the `onSave` DTO):

1. **Project Type** — a picker/segmented control: `complete_rebuild | extension | renovation` (MVP: the three types with National lookup files). Maps to `SuggestCriticalPathRequest.project_type`.
2. **State** (optional) — a picker: NSW + "Other / National fallback". Maps to `SuggestCriticalPathRequest.state`.

Both are appended at the bottom of the form before the submit button, clearly labelled. They are not required — if `projectType` is not set, default to `'complete_rebuild'` (the most complete template) and omit `state` (fallback to National).

## 5. Acceptance Criteria Checklist

### DB & Repository

- [ ] Drizzle migration SQL exists for `order` column in `tasks` table
- [ ] Migration is nullable with no default (existing rows unaffected)
- [ ] `DrizzleTaskRepository.save()` persists `order`
- [ ] `DrizzleTaskRepository.update()` persists `order`
- [ ] `DrizzleTaskRepository.findById()` / `findByProjectId()` return `order`

### Idempotency

- [ ] `stableId(projectId, suggestionId)` utility exists and is pure/deterministic
- [ ] `useCriticalPath.confirmSelected` uses stable IDs on first call and on retry
- [ ] Retry only re-attempts tasks that previously failed; already-created tasks are skipped
- [ ] Re-running `confirmSelected` with the same project + suggestion IDs does not create duplicate DB rows

### Project Creation Flow

- [ ] `ManualProjectEntryForm` renders a `projectType` picker (complete_rebuild / extension / renovation)
- [ ] `ManualProjectEntryForm` renders an optional `state` picker (NSW / Other)
- [ ] `onSave` DTO includes `projectType` and `state`
- [ ] After successful `createProject`, `ManualProjectEntry` transitions to the CriticalPath step (no Alert)
- [ ] CriticalPath step receives correct `projectId` and `SuggestCriticalPathRequest`
- [ ] `CriticalPathPreview` is shown with all tasks pre-selected
- [ ] "Select all / Deselect all" is functional
- [ ] "Add N Tasks to Plan" CTA shows correct count
- [ ] Progress indicator shows "Creating tasks… X / N" while in flight
- [ ] On partial failure, error banner + "Retry" button is shown
- [ ] On "Retry", only failed tasks are retried
- [ ] On completion, the modal closes cleanly
- [ ] A "Skip" affordance is present so users can bypass task seeding

### Quality

- [ ] All unit and integration tests pass
- [ ] `npx tsc --noEmit` — zero TypeScript errors
- [ ] No raw SQL outside of `DrizzleTaskRepository` and migration files
